# データベース運用・保守マニュアル

本ドキュメントは、UVERworld Setlist Archive のデータベース運用に関する標準手順書です。

## 1. バックアップ運用

### 1.1 自動バックアップ
`scripts/backup-db.sh` が Cron によって毎日実行されます。
- **実行時間**: 毎日 03:00 (JST)
- **保存先**: `/var/backups/postgres/`
- **内容**: カスタム形式のダンプ (`-Fc`)、SHA256ハッシュ、Gzip圧縮。

### 1.2 整合性検証
- **日次**: バックアップ完了直後に `pg_restore --list` によるファイル構造の軽量チェックを実施。
- **週次**: 本番負荷を避けるため、**別インスタンス（検証用サーバー）**にて `scripts/backup-db.sh` で生成されたダンプのフルリストア検証を実施。

### 1.3 外部転送と世代管理
- `rsync -avz` を使用し、チェックサム検証を伴う外部バックアップサーバーへの転送。
- 転送先では日付別ディレクトリ (`/backups/YYYY-MM-DD/`) で世代管理を実施。

## 2. 環境間同期 (Prod -> Staging)

本番データを検証環境（Staging）に反映する際は、必ず `scripts/sync-db.sh` を使用します。

### 2.1 実行手順
```bash
./scripts/sync-db.sh /path/to/latest_backup.dump.gz
```

### 2.2 安全対策
- **二段階ガード**: `APP_ENV` (Environment) と `DB_HOST` (Connection) の両方をチェックし、本番環境への誤操作を防止します。
- **メンテナンスモード**: 同期中はStaging環境をメンテナンスモードに移行させます。
- **匿名化**: 同期直後に自動で匿名化クエリが実行されます。
  - `users`: 全情報をダミー化。
  - `comments`: 「（非公開）」へ一律置換。
  - `sessions`: 全クリア。

## 3. 災害復旧 (DR) の5ステップ

万が一のデータ消失時に、最新の状態へ復旧するための手順です。

1. **環境確認**: 復旧対象サーバーの `APP_ENV` および接続先設定が正しいか確認。
2. **バックアップ取得**: 最新の `.dump.gz` および `.sha256` ファイルをバックアップサーバーから取得。
3. **整合性確認**: 
   ```bash
   sha256sum -c backup_filename.dump.sha256
   ```
4. **DBリストア**: 
   ```bash
   pg_restore -d db_name backup_filename.dump.gz
   ```
5. **動作確認**: アプリケーションログを確認し、正常に接続・データ取得ができているか検証。

---

## 4. 監視とトラブルシューティング

- **アラート**: 失敗時（バックアップ失敗、容量不足、転送失敗等）は Discord に即時通知されます。
- **死活監視**: 週次で「成功レポート」が送信されます。このレポートが届かない場合は Cron 自体の停止を疑ってください。
- **ログ確認**: `/var/log/db_backup.log` を参照。
