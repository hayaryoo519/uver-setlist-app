# バージョニング・リリース運用方針

本プロジェクトでは、リリースの管理を明確にするため、セマンティック バージョニング（SemVer）に準拠した運用を行います。

## 1. バージョン番号の付け方
バージョン番号は `v[MAJOR].[MINOR].[PATCH]` の形式で管理します。

| レベル | 変更内容の目安 | 例 |
| :--- | :--- | :--- |
| **MAJOR** | 破壊的変更、大規模なリニューアル、主要機能の廃止 | `v1.5.2` → `v2.0.0` |
| **MINOR** | 新機能の追加、既存機能の大きな改善（後方互換性あり） | `v1.5.2` → `v1.6.0` |
| **PATCH** | バグ修正、軽微なUI調整、文言修正（後方互換性あり） | `v1.5.2` → `v1.5.3` |

## 2. タグ・リリース作成手順
GitHub CLI (`gh`) を使用して、PRマージ後に以下の手順でリリースを作成します。

### ① ブランチの同期
```powershell
git checkout main
git pull origin main
```

### ② 最新バージョンの確認
```powershell
git tag --sort=-v:refname | Select-Object -First 5
```

### ③ リリースの作成
```powershell
# 例: パッチバージョンの更新
& "C:\Program Files\GitHub CLI\gh.exe" release create v1.5.3 --title "v1.5.3 - [簡潔なタイトル]" --notes "[変更内容の要約]"
```

## 3. 運用ルール
- **タグの打機**: タグは必ず `main` ブランチの最新コミットに対して作成します。
- **リリースノート**: リリースノート（Notes）には、ユーザーに影響のある変更点を簡潔に記載します。
- **4桁バージョンの廃止**: 以前使用していた 4 桁のバージョン（`v1.5.1.2` 等）は、互換性維持のため過去分は残しますが、今後は 3 桁に統一します。
